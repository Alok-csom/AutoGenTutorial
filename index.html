Installation

Install AutoGen Studio into your hw6155 environment

There are four sub-steps here that must happen in order: verify Python → create the virtual environment → activate it → install packages. You cannot skip or reorder them.

a
Verify Python is installed first Before anything else, confirm Python 3.10 or later is available on your machine: 
python3 --version
# Expected: Python 3.10.x or higher
python --version
# Expected: Python 3.10.x or higher
If you see "Python was not found" or "not recognized", install Python from python.org/downloads and check "Add Python to PATH" on the first screen of the installer. Then close and reopen PowerShell before continuing.
Windows users — use python, not python3. On Windows, python3 often returns "Python was not found" even when Python is correctly installed, because the standard Windows installer only registers the python command. Every command in this tutorial's PowerShell tab already uses python for this reason — if you're following along and adapting a macOS command yourself, remember to drop the 3 on Windows.
b
Navigate to hw6155 and create the virtual environment The virtual environment must be created before it can be activated — Activate.ps1 does not exist until this step runs. You only need to do this once per project. 
cd hw6155
python3 -m venv .venv
# Creates the .venv folder with its own Python and pip
Set-Location hw6155
python -m venv .venv
# Creates the .venv folder with its own Python and pip
This creates a .venv folder inside hw6155. On Windows it contains .venv\Scripts\Activate.ps1. On macOS it contains .venv/bin/activate. Nothing is installed yet — the next step activates it.
c
Activate the virtual environment Activation tells your terminal to use the packages inside .venv instead of the system-wide Python. You need to do this every time you open a new terminal window. 
source .venv/bin/activate
# Prompt changes to: (.venv) $
.venv\Scripts\Activate.ps1
# Prompt changes to: (.venv) PS C:\...>
Once activated, your prompt shows (.venv) at the start. This confirms all pip install commands will install into the isolated environment, not globally.
d
Install AutoGen Studio and extensions Install everything in one single command so pip can resolve compatible versions across all packages at once. Installing them separately causes version conflicts — for example, autogenstudio 0.4.x requires autogen-agentchat below 0.6, but a separate install pulls in 0.7.x which breaks things. 
# Install all packages together so pip picks compatible versions
pip install \

  "autogen-agentchat>=0.4.9.2,<0.6" \
  "autogen-core>=0.4.9.2,<0.6" \
  "autogen-ext[web-surfer,magentic-one,openai,azure]>=0.4.2,<0.6"

# Install the Playwright robot browser (see note below)
playwright install --with-deps chromium

pip uninstall autogenstudio autogen-agentchat autogen-core autogen-ext -y
pip uninstall autogenstudio autogen-agentchat autogen-core autogen-ext -y
Then run the combined install command above.
Why Playwright and not your regular browser?
The WebSurfer agent needs to control a browser through code — opening tabs, typing searches, clicking links, scrolling, and reading page text back into Python. Your regular Chrome or Edge cannot be driven this way. Playwright installs a special version of Chromium with automation hooks built in that the agent operates like a robot.

Your regular browser Playwright Chromium
Who controls itYou (mouse & keyboard)The AutoGen agent (code)
Agents can drive it❌ No✅ Yes
Visible on screenAlwaysOptional (headless mode)
PurposeGeneral browsingAgent web automation

By default the agent runs Chromium invisibly in the background (headless mode). In Step 5 you can switch to headless=False to watch a real browser window open on your screen and the agent navigate by itself in real time — a great classroom demonstration.
You do NOT need a Google Search API, a Bing Search API, or any separate search API key for this tutorial. WebSurfer doesn't call a search API at all — it opens a real browser (via Playwright), navigates to a search engine's normal web page (Bing, by default, at www.bing.com), types into the search box, clicks results, and reads the resulting page the same way a human would. There's no API to sign up for and no extra key to add to your .env file — your Azure AI Foundry key is the only credential this entire tutorial needs. If a Google API setup screen shows up while you're researching this on your own, you've landed on a different (unrelated) project — close that tab and come back here.

The order matters every session. Steps b and c are one-time setup. But every time you open a new terminal to work on hw6155, you must run step c (activate) before running any pip, autogenstudio, or python commands — otherwise they run against your global Python instead of the isolated environment.
Windows execution policy error — if you see "running scripts is disabled on this system", you have three options. Try them in order:

Option 1 — Fix the policy for your user account (recommended):
Run this in PowerShell, type Y when prompted, then retry Activate.ps1:

# Type Y and press Enter when prompted
# Then run the activation again:
.venv\Scripts\Activate.ps1

Option 3 — Use Command Prompt instead of PowerShell (always works):
In VS Code, click the ˅ dropdown arrow next to the + in the terminal panel and choose Command Prompt. Then run:

:: Note: .bat not .ps1 — no execution policy restrictions
:: Prompt changes to: (.venv) C:\...>

Team 1

Understand and configure the WebSurfer Team

The Web Agent Team (Operator) gallery template already has three agents wired together with working prompts. The goal in this step is not to rewrite those prompts from scratch — it's to understand what each one already does, point all three at your Azure model, and only adjust the parts you actually need to change.

Don't overwrite the selector prompt with a plain summary instruction. The selector prompt on this template is not a normal agent system message — it's the text the team orchestrator uses to decide who speaks next, and it relies on built-in template variables ({roles}, {participants}, {history}) that Studio fills in automatically every turn. Replacing it with a flat "write a summary and say TERMINATE" instruction breaks the routing logic and duplicates a job that assistant_agent already does correctly.
🌐

websurfer_agent

Browser agent

Controls a real Chromium browser — navigates, clicks, scrolls, and extracts text from any web page. Reports raw findings back to the team.

✅

assistant_agent

Verifier / summarizer

Checks whether the task is actually done. If not, says what's left and "keep going." If done, writes the final summary and says TERMINATE — this is the only place TERMINATE belongs.

🙋

user_proxy

Human-in-the-loop

Stands in for you. Selected automatically when the team needs human feedback, a preference, or gets stuck — not something you write a system prompt for.

User task
→
websurfer_agent
browses web
↔
assistant_agent
verifies / "keep going"
→
TERMINATE + final answer

user_proxy joins this loop only if the team needs human input — it isn't on the happy-path flow above.

Follow these steps in the Studio browser UI:

1
Start from the Gallery

Click Gallery in the left sidebar, then the Teams tab. Find the Web Agent Team (Operator) template card and click Use Template (or duplicate it, the same way you duplicated the Azure model in Step 3). This opens the pre-wired team with all three agents — websurfer_agent, assistant_agent, user_proxy — already on the canvas, with their default prompts intact.

4
Read — don't rewrite — the selector prompt

Open the Team settings (click the outer canvas, not an agent) and find the Selector Prompt field. This is the default text from your gallery template:

Default selector prompt (leave as-is for now)
You are the coordinator of a role play game. The following roles are
available: {roles}. Given a task, the websurfer_agent will be tasked to
address it by browsing the web and providing information. The
assistant_agent will be tasked with verifying the information provided
by the websurfer_agent and summarizing the information to present a
final answer to the user. If the task needs assistance from a human
user (e.g., providing feedback, preferences, or the task is stalled),
you should select the user_proxy role to provide the necessary
information.

Read the following conversation. Then select the next role from
{participants} to play. Only return the role.

{history}

Read the above conversation. Then select the next role from
{participants} to play. Only return the role.

{roles}, {participants}, and {history} are template variables — Studio substitutes the real role descriptions, agent names, and live conversation into these placeholders on every single turn. This is what lets one short prompt route between three agents intelligently as the conversation unfolds. There's no need to touch this for the base tutorial — it already tells the orchestrator exactly when to pick websurfer_agent, assistant_agent, or user_proxy.

What each variable actually gets replaced with:

VariableWhat it resolves to
{roles}A list of every agent's name + description pulled straight from each agent's config — the same description you typed when you set up that agent. For this team it expands to the websurfer_agent, assistant_agent, and user_proxy descriptions.
{participants}Just the names of the agents currently on the team, e.g. websurfer_agent, assistant_agent, user_proxy. This is the exact list the selector LLM is told to choose one value from.
{history}The full conversation so far — every message exchanged between agents and the user up to this point — pasted in as plain text. This is how the selector "knows" what already happened and decides who should logically speak next.

Worked example — imagine the conversation so far is just the user's task: "What is the capital of France?" Before asking the selector LLM anything, Studio fills in the template like this:

What the selector LLM actually receives (variables already filled in)
You are the coordinator of a role play game. The following roles are
available: websurfer_agent - Browses the web to find information,
assistant_agent - Verifies findings and summarizes the final answer,
user_proxy - Represents the human user. Given a task, the
websurfer_agent will be tasked to address it by browsing the web...

Read the following conversation. Then select the next role from
websurfer_agent, assistant_agent, user_proxy to play. Only return
the role.

user: What is the capital of France?

Read the above conversation. Then select the next role from
websurfer_agent, assistant_agent, user_proxy to play. Only return
the role.

The selector LLM reads this and replies with a single word — websurfer_agent — because nothing has been researched yet. On the next turn, {history} will have grown to include websurfer_agent's browsing results, and the selector will likely reply assistant_agent instead, since there's now something to verify and summarize. This re-evaluation happens fresh on every single turn — the selector never "remembers" its last choice, it just re-reads the (now longer) history each time.

5
Understand where TERMINATE actually comes from

Click assistant_agent and look at its System Message — this is the default, and it's already correct:

assistant_agent's default system message (leave as-is for now)
You are a task verification assistant who is working with a web surfer
agent to solve tasks. At each point, check if the task has been
completed as requested by the user. If the websurfer_agent responds
and the task has not yet been completed, respond with what is left to
do and then say 'keep going'. If and only when the task has been
completed, summarize and present a final answer that directly
addresses the user task in detail and then respond with TERMINATE.

This is the only place TERMINATE should be written. assistant_agent is doing two jobs at once: verifying whether websurfer_agent's findings actually answer the task, and only producing the final summary + TERMINATE once that check passes. If you separately add a "write a summary and say TERMINATE" instruction somewhere else, you now have two agents that think they're responsible for ending the conversation, which is exactly what was confusing earlier.

6
Verify the team-level termination condition

Still in Team settings, confirm the termination condition includes TextMentionTermination with the keyword TERMINATE and MaxMessageTermination set to 12. This is the safety net that stops the team if assistant_agent's own TERMINATE never fires.

7
Save the team

Click Save Team (top right of canvas). Your team is now saved to the ./studio_data database.

If you do want to customize behavior later (Step 8 has exercises for this), edit assistant_agent's system message directly rather than adding a second, competing set of instructions elsewhere. Keep the verify → "keep going" → summarize → TERMINATE structure, and just change tone, format, or strictness within that structure.
How to actually provide input when user_proxy is selected — and what NOT to do.

When the selector picks user_proxy mid-run, the Playground will pause and let you type a message directly into the chat box, right in the same ongoing session. Type your input and send it — the run continues from where it paused, with the full conversation history intact.

Do not click "Cancel Run" to inject input. Cancelling a run does not pause or resume it — it ends the run entirely and discards {history}. If you then paste a message and start again, the selector is reading a conversation that contains only your new message — none of what websurfer_agent or assistant_agent already produced. This typically causes the whole team to restart from the beginning, including agents whose description says they should "only speak at the start of a new task," because as far as the team object is concerned, it genuinely is a new task.

If you build a team where you want a human to weigh in partway through (for example, approving a draft before a final agent finalizes it), make sure user_proxy (or another UserProxyAgent) is an actual participant on the team, and write its description to say clearly when it should be selected — the same way you'd write a description for any other agent. That's the only way to get a real pause-and-resume instead of a destructive restart.
Adding agents by drag and drop: From the Component Library, drag any agent type onto the canvas. A connector line automatically appears between agents. Drag it to the team orchestrator to wire the message flow. You can also drag agents off the canvas to remove them.

Team 2

Build the Deep Research Team from scratch

The Deep Research Team uses a SelectorGroupChat orchestration — an LLM dynamically picks which of four agents speaks next. This time you're building the team yourself rather than starting from a gallery template, which means you are responsible for writing a selector prompt that uses the template variables correctly — the same {roles} / {participants} / {history} mechanics you saw in Step 4.

Don't hardcode the routing order in the selector prompt. A tempting shortcut is to write rules like "ResearchPlanner speaks first, then WebResearcher, then Critic, then ReportWriter" directly into the prompt. This works for the happy path but breaks the moment the conversation needs to loop back — for example if Critic asks a follow-up and WebResearcher needs to go again before Critic re-checks. The Web Agent Team template you studied in Step 4 doesn't hardcode an order at all — it describes each agent's role via {roles} and lets the selector LLM reason about what should happen next given the actual conversation in {history}. You'll do the same thing here.

Create a new team

Name: ResearchPlanner

Description — this is what fills {roles}, write it carefully:

ResearchPlanner description
Breaks a research topic into 3-5 specific, answerable sub-questions
and assigns them one at a time to WebResearcher. Should speak first
when a new topic arrives, and again later only if the Critic identifies
a gap that needs a new sub-question.

System Message — governs how it actually behaves once selected:

ResearchPlanner system message
You are a senior research strategist. When given a topic, break it into
3-5 specific, answerable sub-questions. Assign each to WebResearcher.
Be concise and precise.
Description vs system message — different jobs. The Description field is what the selector LLM reads via {roles} to decide whether to pick this agent next. The System Message is what the agent itself reads to decide what to actually say once picked. A vague description ("does research planning") gives the selector nothing to reason with; a specific one (as above) tells it exactly when this agent should and shouldn't be chosen.

6
Write a selector prompt that uses all three variables

Click the outer canvas (Team settings). Find the Selector Prompt field. Instead of listing a fixed turn order, write a prompt that leans on {roles} to describe the team and lets the selector reason using {history} — this is the same structure as the Web Agent Team's default prompt from Step 4:

Selector prompt — uses {roles}, {participants}, {history}
You are coordinating a research team. The following roles are
available: {roles}

Read the conversation so far and decide which single role should act
next, based on what each role's description says it is responsible
for and what has or hasn't happened yet in the conversation.

Conversation so far:
{history}

Select the next role from {participants} to act. Only return the
role name, nothing else.

Notice what changed: there is no hardcoded "ResearchPlanner speaks first, then WebResearcher..." rule anywhere. The routing logic now lives entirely in the agent descriptions you wrote in steps 2–5 — exactly the same division of labor as the real Web Agent Team template. If Critic asks a follow-up, the selector can send the conversation back to WebResearcher or ResearchPlanner because their descriptions say that's a valid reason to pick them, instead of being stuck following a fixed sequence that has no "go back" option.

7
Configure termination conditions

In Team settings, add two termination conditions:

① TextMentionTermination — keyword: TERMINATE
② MaxMessageTermination — max messages: 20

Set the combination logic to OR (stop when either condition is met). As in Step 4, TERMINATE should only ever appear in ReportWriter's system message — it's the one agent whose job is to end the conversation, the same way assistant_agent was the only place TERMINATE belonged in the WebSurfer Team.

8
Save the team

Click Save Team. Your four-agent Deep Research Team is now saved and ready to run.

SelectorGroupChat vs RoundRobinGroupChat: With RoundRobin, agents always take turns in fixed order. With SelectorGroupChat, the Selector LLM reads {roles} and {history} together and decides who is most appropriate to speak next — enabling dynamic research depth, conditional retries, and loops back to an earlier agent when needed, none of which a fixed turn order or a hardcoded rule list can do.

Common issues:
• Model connection error — verify your Azure JSON in the Model Library (Step 3). Check that the endpoint URL ends in .openai.azure.com/ with a trailing slash.
• Browser not found — run playwright install --with-deps chromium in your active .venv terminal.
• Session hangs after 10+ messages — the MaxMessageTermination limit was hit. Increase it in Team settings or rephrase your task to be more specific.
429 Too Many Requests — rate limit error: If you see a openai.RateLimitError: Error code: 429 in the Playground, your Azure GPT-4o deployment has hit its tokens-per-minute (TPM) limit. The WebSurfer agent makes many rapid API calls as it browses — each screenshot, navigation decision, and summary is a separate call. Try these fixes in order:

1 — Increase your deployment quota in Azure (recommended):
Go to ai.azure.com → your project → Deployments → click your GPT-4o deployment → Edit → increase Tokens per minute to 80K or higher. Low quotas like 8K–10K are too small for WebSurfer.

2 — Lower the message limit to reduce API calls:
In Team Builder, open the Web Agent Team settings and reduce MaxMessageTermination from 12 down to 6. Fewer turns = fewer API calls per session.

3 — Use a narrower, more specific task:
Broad tasks like "research AI in business" cause the agent to browse many pages in many turns. Try something focused that can be answered in 2–3 turns, for example:
"What is the official Microsoft AutoGen GitHub URL? Find the current star count."

4 — Wait 60 seconds and retry:
Azure rate limits reset every minute. If you hit the limit, wait a moment and try again with a simpler task first to confirm the connection is working.

Export

Export your team to Python code

Any team you build in Studio can be exported as a Python JSON configuration and loaded by the AutoGen Python API. This lets you move from no-code prototyping to production code with one click.

a
Export the team JSON from Studio Open your team in Team Builder. Toggle JSON mode (the { } button at the top right of the canvas). Copy the entire JSON, or click Download to save it as a file — e.g., websurfer_team.json — in your hw6155 folder.
b
Load and run the team from Python Create a file run_team.py in hw6155 with the following code:
Python — run_team.py
import asyncio, json
from autogen_agentchat.teams import TeamManager
from autogen_agentchat.ui import Console

async def main():
    # Load the team exported from AutoGen Studio
    with open("websurfer_team.json") as f:
        team_config = json.load(f)

    team = TeamManager.load_component(team_config)

    task = "Research recent AI applications in healthcare for 2024-2025."
    stream = team.run_stream(task=task)
    await Console(stream)

asyncio.run(main())
c
Run the exported team 
python3 run_team.py
python run_team.py
The team runs exactly as it does in the Playground — same agents, same model, same orchestration — but now from the command line.
The exported JSON is exactly the component format used by the AutoGen Python API. You can check it into Git, share it with teammates, or load it in a production application — the Studio UI and Python code are fully interchangeable.
